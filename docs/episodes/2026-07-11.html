The attention backend you never chose

Alan: Okay, quick one — but keep it, because it might make your inference faster for free. Hugging Face published a profiling post this week comparing the attention backends in PyTorch. And the punchline is that the default choice you never made is probably costing you.

Ada: Set it up. When a model runs attention, PyTorch has this one call — scaled dot-product attention — and underneath it there are several different implementations it can dispatch to.

Alan: Right. There's a naive one, a "math" one, and then the fast fused ones — Efficient, Flash, cuDNN. And most people never pick. They call the function and take whatever they get.

Ada: And here's the history that makes the numbers make sense. The old way, every step of attention is its own GPU operation — multiply, scale, softmax, multiply again — and each one launches separately and shuffles data back and forth to memory. The breakthrough a few years ago, FlashAttention, was to fuse all of that into a single kernel that never writes the big intermediate matrix to memory at all.

Alan: And the profiling shows exactly that gap in kernel counts. Hugging Face reports the naive path launches five or six GPU kernels per forward pass. The math backend launches twenty. And Flash, Efficient, and cuDNN? One fused kernel each.

Ada: Twenty versus one. That's the whole story of the last few years of systems work in a single ratio.

Alan: And it shows up in wall-clock. On an A100, they measure the math backend running about three point seven times slower than even the naive in-place version. Three-point-seven-x slower, just from a backend choice nobody consciously makes.

Ada: There's one lovely wrinkle in there, though, that I don't want to skip, because it breaks the intuition. Flash is the fastest backend. And Flash runs at about thirteen percent occupancy — meaning it's leaving something like eighty-seven percent of the GPU's threads idle.

Alan: Wait — the fastest one is barely using the chip?

Ada: The fastest one is barely using the chip. Which is the lesson, really. We're trained to think "utilization high, good." But occupancy isn't speed. Flash wins by not touching memory, not by keeping every core busy. A kernel that saturates the GPU doing wasteful memory traffic loses to one that leaves cores idle and never leaves the chip.

Alan: And the reason that's the right trade is that attention is memory-bound, not compute-bound. The chip can multiply numbers faster than it can fetch the numbers to multiply. So the real bottleneck is data movement, and a kernel that keeps everything on-chip wins even with the math units sitting idle.

Ada: Which inverts the instinct completely. You optimize the trip to memory, not the arithmetic.

Alan: So the practical takeaway for anyone serving a model — go look at which backend you're actually dispatching to, because the default isn't always the fused one, and the gap is multiples, not percentages. In PyTorch you can force it, or at least print which one you got.

Ada: Small thing. Keep it forever.
